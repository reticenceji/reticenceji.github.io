创建一个Widget

可以用类C语言风格的API,如下

#![allow(unused)]
fn main() {
let app = Application::new(None, ApplicationFlags::FLAGS_NONE);
app.set_application_id(Some("org.gtk.example"));

let window = ApplicationWindow::new(&app);
window.set_title(Some("My GTK App"));
}

但是更优雅的做法是使用“建造者模式”风格的API.

#![allow(unused)]
fn main() {
let app = Application::builder()
	.application_id("org.gtk.example")
	.build();

let window = ApplicationWindow::builder()
        .application(&app)
        .title("My GTK App")		
        .build();
}

callback function :star:

比如,我们要为``button的click`动作,关联一个回调函数.在C语言中是这样

g_signal_connect (G_OBJECT (button), "clicked", G_CALLBACK (click_button), NULL);

而在Rust的绑定中会显得简单,毕竟C并不是面向对象的语言.而Rust是具有面向对象的特性的.

#![allow(unused)]
fn main() {
// button: Button
button.connect_clicked(move |button| {
        // Set the label to "Hello World!" after the button has been clicked on
        button.set_label("Hello World!");
    });
}

Rust风格的绑定回调函数是connect_<method>, 然后参数是一个函数. 这种简单的函数用closure就挺好的. 也可以顺便查看一下源码如下,不难发现这就是一个对C语言函数的封装. connect_raw就是g_signal_connect.

#![allow(unused)]
fn main() {
fn connect_clicked<F: Fn(&Self) + 'static>(&self, f: F) -> SignalHandlerId {
        unsafe extern "C" fn clicked_trampoline<P: IsA<Button>, F: Fn(&P) + 'static>(
            this: *mut ffi::GtkButton,
            f: glib::ffi::gpointer,
        ) {
            let f: &F = &*(f as *const F);
            f(Button::from_glib_borrow(this).unsafe_cast_ref())
        }
        unsafe {
            let f: Box_<F> = Box_::new(f);
            connect_raw(
                self.as_ptr() as *mut _,
                b"clicked\0".as_ptr() as *const _,
                Some(transmute::<_, unsafe extern "C" fn()>(
                    clicked_trampoline::<Self, F> as *const (),
                )),
                Box_::into_raw(f),
            )
        }
    }
}

一个需要注意的问题是, callback function默认是和主线程一起的. 也就是说, 如果回调函数特别复杂,那么整个界面就卡住了. 这是我们不希望看见的.

我原本以为只需要把回调函数变成一个新建的线程就可以解决问题,像下面这样. 但是编译器会告诉你“NonNull<GObject> cannot be shared between threads safely within gtk4::Button, the trait Sync is not implemented for NonNull<GObject> required because of the requirements on the impl of Send for &gtk4::Button”. 其实,GTK is not thread-safe. Accordingly, none of this crate’s structs implement Send or Sync. 所以就不要想着简单的把GTK的widget传入新建的线程了,都放在主线程渲染吧. 不过这样也不是完全不能解决问题, 你可以把和widget无关的复杂逻辑放进线程中,再和主线程通信一下(也许可以吧)

#![allow(unused)]
fn main() {
button.connect_clicked(move |button| {
        // Set the label to "Hello World!" after the button has been clicked on
        thread::spawn(move||{
            button.set_label("Hello World!"); // can be outside the child thread
            loop{}
        });
    });
}

还有另外一种解决方案, 参考https://gtk-rs.org/gtk4-rs/stable/latest/book/main_event_loop.html. 注意

  • spawn_local 使用的感觉就像创建了一个子线程, 但是实际上还是主线程. 这样有一个好处就是避免了跨线程传递数据的问题, 可以把GObject传给闭包.
  • 所以我们如何选择创建一个子线程还是spawn_local呢?
    • If you have async functions for your IO-bound operations at your disposal, feel free to spawn them on the main loop.
    • If your operation is computation-bound or there is no async function available, you have to spawn threads.
  • channel可以给receiver attach一个处理传递来数据的闭包. 虽然我不知道这个是怎么实现的, 让我自己实现会用轮询. 不过这个功能确实很好用.
  • clone!宏可以避免写很多东西, 但是我还不太了解.所以我都是手动clone.
#![allow(unused)]
fn main() {
let (sender, receiver) = MainContext::channel(PRIORITY_DEFAULT);
    // Connect to "clicked" signal of `button`
    button.connect_clicked(move |_| {
        let main_context = MainContext::default();
        // The main loop executes the asynchronous block
        main_context.spawn_local(clone!(@strong sender => async move {
            // Deactivate the button until the operation is done
            sender.send(false).expect("Could not send through channel");
            timeout_future_seconds(5).await;
            // Activate the button again
            sender.send(true).expect("Could not send through channel");
        }));
    });

    // The main loop executes the closure as soon as it receives the message
    receiver.attach(
        None,
        clone!(@weak button => @default-return Continue(false),
                    move |enable_button| {
                        button.set_sensitive(enable_button);
                        Continue(true)
                    }
        ),
    );
}

常用的Widget functions

  • destroy: 通常只被 top-level widgets 使用. It is usually only used to destroy dialog windows and to implement menu items that quit the application.
  • set_width_request和set_height_request: 顾名思义, 设置widget的宽和高
  • set_sensitive(bool): 让widget处于激活/非激活状态.

详细请参考: https://gtk-rs.org/gtk4-rs/stable/latest/docs/gtk4/prelude/trait.WidgetExt.html

LayoutManager

LayoutManager的主要目的是为了让parent widget 有多个children. 在GTK有两种类型的container, 用于 laying out children and decorators and those that add some sort of functionality beyond positioning to a child.看了一下,大致有

  • BinLayout is a LayoutManager subclass useful for create “bins” of widgets.BinLayout will stack each child of a widget on top of each other.
  • BoxLayout is a layout manager that arranges children in a single row or column.
  • CellLayout is an interface to be implemented by all objects which want to provide a TreeViewColumn like API for packing cells, setting attributes and data funcs.
  • CenterLayout is a layout manager that manages up to three children.
  • ConstraintLayout is a layout manager that uses relations between widget attributes, expressed via Constraint instances, to measure and allocate widgets.
  • FixedLayout is a layout manager which can place child widgets at fixed positions.
  • GridLayout is a layout manager which arranges child widgets in rows and columns.

所以我们需要如何使用LayoutManager呢?

  1. 每一个widget只可以属于一个LayoutManger, 我们通过WidgetExt::set_layout_manager<P: IsA<LayoutManager>>(&self, layout_manager: Option<&P>);设置widget的LayoutManager.

  2. 我们好像不直接使用LayoutManager.

    LayoutManagerWidget
    BinLayoutBin
    BoxLayoutgtk4::Box(这个容易引起误会)
    CenterLayoutCenterBox
    ConstraintLayoutConstraint
    FixedLayoutFixed
    GridLayoutGrid

那么我们可以这样:

#![allow(unused)]
fn main() {
let container = CenterBox::builder().build();

let b1 = Button::builder().label("Center").build();
let b2 = Button::builder().label("Start").build();
let b3 = Button::builder().label("End").build();
container.set_center_widget(Some(&b1));
container.set_start_widget(Some(&b2));
container.set_end_widget(Some(&b3));

// button.set_sensitive(false)
window.set_child(Some(&container));
}

继承关系

使用container碰到的一个问题, 写下面的代码时候, 编译器提示说container不是Widget(the trait IsA<gtk4::Widget> is not implemented for BoxLayout), set_child这个函数只能接受Widget的参数. 的确, LayoutManager不是Widget, 可以参考GTK的Class继承关系. 

#![allow(unused)]
fn main() {
window.set_child(Some(&container));
}

我们知道GTK的Class继承关系. 那么我们在Rust中是如何体现这个继承关系的呢?就是这个IsA<T>. IsA是一个Unsafe Paradigm Trait, 并且继承于ObjectType+AsRef<T>+'static. 他没有方法, 也就是说只是一个标记.

#![allow(unused)]
fn main() {
pub unsafe trait IsA<T: ObjectType>: ObjectType + AsRef<T> + 'static { }
}

复习一下Rust知识, A trait is unsafe when at least one of its methods has some invariant that the compiler can’t verify. 继承的意思是说, impl IsA的类型必须同时 impl ObjectType+AsRef<T>+'static.

那么光靠这个标记我们是怎么确定类的继承关系的呢? 以Button为例, Button满足如下IsA<T>. 代码的实现借助了宏.

impl IsA<Accessible> for Button
impl IsA<Actionable> for Button
impl IsA<Buildable> for Button
impl IsA<Button> for LinkButton
impl IsA<Button> for LockButton
impl IsA<Button> for ToggleButton
impl IsA<ConstraintTarget> for Button
impl IsA<Widget> for Button

我们有时可能会想要创建自定义的类型, 如下, 我们创建了一个自定义Button.

fn main() {

gio::ListStore

我们经常有需求需要把一些信息按列显示, 这就是List派上用场的地方. 虽然有更简单的ListBox, 但是建议使用:

  • The model holds our data, filters it and describes its order.
  • The list item factory defines how the data transforms into widgets.
  • The view specifies how the widgets are then arranged.

参考https://gtk-rs.org/gtk4-rs/stable/latest/book/lists.html

// SignalListItemFactory: 
let factory = gtk::SignalListItemFactory::new();

    list_item.set_child(Some(&label));

factory.connect_bind(move |_, list_item| {

    let client_object = list_item
    .item()
    .expect("The item has to exist.")
    .downcast::<list::ClientObject>()
    .expect("The item has to be an `ClientObject`.");

    // Get `i32` from `ClientObject`
    let no = client_object
    .property("no")
    .expect("The property needs to exist and be readable.")

    let addr = client_object
    .property("addr")
    .expect("The property needs to exist and be readable.")

    // Get `Label` from `ListItem`
    let label = list_item
    .child()
    .expect("The child has to exist.")
    .downcast::<gtk::Label>()
    .expect("The child has to be a `Label`.");

// 这是单选

需要注意的是, gio::ListStore only accepts GObjects. 所以我们需要创建一个GObject的子类来存储数据. 

#![allow(unused)]
fn main() {
use glib::{ParamFlags, ParamSpec, Value};
use gtk::glib;
use gtk::prelude::*;
use gtk::subclass::prelude::*;
use once_cell::sync::Lazy;
use std::cell::Cell;
use std::cell::RefCell;
use glib::Object;

// Object holding the state
#[derive(Default)]
pub struct _ClientObject {
    no: Cell<i32>,
    addr: RefCell<String>,
}

// The central trait for subclassing a GObject
#[glib::object_subclass]
impl ObjectSubclass for _ClientObject {
    const NAME: &'static str = "MyGtkAppClientObject";
    type Type = ClientObject;
    type ParentType = glib::Object;
}

// Trait shared by all GObjects
impl ObjectImpl for _ClientObject {
    fn properties() -> &'static [ParamSpec] {
        static PROPERTIES: Lazy<Vec<ParamSpec>> = Lazy::new(|| {
            vec![
                ParamSpec::new_int(
                    // Name
                    "no",
                    // Nickname
                    "no",
                    // Short description
                    "no",
                    // Minimum value
                    i32::MIN,
                    // Maximum value
                    i32::MAX,
                    // Default value
                    0,
                    // The property can be read and written to
                    ParamFlags::READWRITE,
                ),
                ParamSpec::new_string("addr", "addr", "addr", None, ParamFlags::READWRITE),
            ]
        });
        PROPERTIES.as_ref()
    }

    fn set_property(&self, _obj: &Self::Type, _id: usize, value: &Value, pspec: &ParamSpec) {
        match pspec.name() {
            "no" => {
                let input_no = value.get().expect("The value needs to be of type `i32`.");
                self.no.replace(input_no);
            },
            "addr" => {
                let input_addr = value.get().expect("The value needs to be of type `String`.");
                self.addr.replace(input_addr);
            }
            _ => unimplemented!(),
        }
    }

    fn property(&self, _obj: &Self::Type, _id: usize, pspec: &ParamSpec) -> Value {
        match pspec.name() {
            "no" => self.no.get().to_value(),
            "addr" => {self.addr.borrow().clone().to_value()}
            _ => unimplemented!(),
        }
    }
}

glib::wrapper! {
    pub struct ClientObject(ObjectSubclass<_ClientObject>);
}

impl ClientObject {
    pub fn new(no: i32,addr: String) -> Self {
        Object::new(&[("no", &no),("addr",&addr)]).expect("Failed to create `ClientObject`.")
    }   
}
}

闭包

因为想让回调函数中的复杂逻辑以子线程的方式完成, 所以有了下面的代码. 但是编译器提示你

cannot move out of tcp_w, a captured variable in an Fn closure move out of tcp_w

#![allow(unused)]
fn main() {
{
    ...
    let tcp_w = TCP_W.clone();
    time_button.connect_clicked(move |_| {
        thread::spawn(move || {
            if let Some(tcp_w) = tcp_w.lock().unwrap().as_mut() {
                tcp_w.request_time();
            };
        });
    });
    ...
}
}

回顾Rust函数闭包, 有三种类型, 将f(a,b,c)变成

  • Fn::call(&f,a,b,c) 可以调用多次
  • FnMut::call(&mut f,a,b,c)
  • FnOnce::call(f,a,b,c) 会转移方法接收者的所有权, 也就是说只能调用一次

所以, 像上面的错误代码, thread就调用一次之后, tcp_w的生命就结束了.不能调用第二次了.

另外, 为什么需要move关键字呢? 如果我们不使用move关键字, 那么当函数结束的时候, tcp_w就被drop了,但是我们的函数闭包会一直需要tcp_w(因为我们怎么知道什么时候按钮会被点击呢?).所以, 要么通过move直接吧ownership也移动进闭包, 要么传递一个生命周期为‘static的引用.

GObject

A GObject (or glib::Object in Rust terms) is a reference-counted, mutable object.

NonNull<GObject> cannot be sent between threads safely within [closure@src/main.rs:237:35: 255:22], the trait Send is not implemented for NonNull<GObject>

如果你试图把gtk中的类传递给别的线程, 会给出如上的错误. 错误的原因是GObject并不能安全的跨线程传输. 所以在编程的时候, 可以把组件渲染的逻辑放在主线程, 其他逻辑放在子线程, 用一些方法传递数据.
