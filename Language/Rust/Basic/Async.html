Reticence's Notes

异步编程

Generator&Future

Generator和Future是后面异步的基础。不过他们并没有什么特殊之处。先来看

#![allow(unused)]
fn main() {
pub trait Generator<R = ()> {
    type Yield;
    type Return;
    fn resume(
        self: Pin<&mut Self>, 
        arg: R
    ) -> GeneratorState<Self::Yield, Self::Return>;
}
}

不过，不像一般的trait，我们自己去为我们自己定义的struct实现它。我们是用一个写法非常类似闭包的东西自动生成一个实现了Generator的struct，而这种写法和闭包的区别在于如果是否有yield。比如下面这个例子，就是一个类似于Iterator的Generator。

#![allow(unused)]
#![feature(generators, generator_trait)]

fn main() {
use std::{ops::Generator, pin::Pin};
// 这个是返回值泛型，看到的时候楞了一下
pub fn count() -> impl Generator<Yield = u64, Return = ()> {
    || {
        let mut x = 0;
        loop {
            x += 1;
            yield x;
        }
        return ();
    }
}
}
  • 像上面这样，我们只关心Yield的类型，不关心Return的类型，就跟Iterator很像。resume = next
  • 又或者，我们只关心Return的类型，不关心Yield的类型，就跟Future很像，看下面的Future的定义，resume = poll
#![allow(unused)]
fn main() {
// 可以简单的将Future理解为下面这样
// trait SimpleFuture {
//     type Output;
//     fn poll(&mut self, wake: fn()) -> Poll<Self::Output>;
// }
trait Future {
    type Output;
    fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
}

enum Poll<T> {
    Ready(T),
    Pending,
}
}
  • Future的操作（函数）只有一个，就是poll，字面意思就是轮询。轮询的结果可能是Pending，表示还没有算完；也可以是Ready(T)，返回结果。
  • poll还附带了一个wake函数，可以将它理解为回调函数，我们每次poll发现Pending，就注册回调函数。当回调函数被调用时，就会再次执行poll。

单看上面的代码，Future也没有什么特殊之处。事实上，他是异步计算单元的抽象，而具体异步计算单元的执行和调度才是异步发挥其高效性的地方。

Futures are a concept that abstracts over how code is run. By themselves, they do nothing.

async/await

用async/await进行异步代码的开发是现在很流行的方式。如下面的例子

use futures::executor::block_on;

async fn compute() -> u64 {
    println!("Computing...");
    114
}

async fn hello_world() -> u64 {
    let a = compute().await;
    let b = compute().await;
    a + b
}

fn main() {
    // let a: impl Future<Output = u64> = hello_world();
    assert_eq!(228, block_on(hello_world()));
}

  1. 写起来的逻辑和同步代码一模一样。不过，事实上async函数的参数可能是在其他线程执行的，我们要保证线程安全。

  2. async是什么意思呢？他用于定义异步函数/异步块(之后称为async块)。上面定义的computer和hello_world函数的实际返回值是一个实现了Future的struct（也可以简单的说返回了一个Future），也就是一个异步计算单元。

  3. await是什么意思呢？他在逻辑上，就是去执行Future并等待他完成。但是具体怎么等呢，这个是由异步运行时决定的。你可以简单理解为future.await在逻辑上为下面的代码

    #![allow(unused)]
fn main() {
loop {
  match future.poll() {
    Pending => yield(),  // 让出CPU
    Ready(r) => break r; // 返回结果
  }
}
}

How async construct a Future

更准确的说法是，async是如何创造了一个匿名的struct，实现了Future的呢？参考tokio的文档https://tokio.rs/tokio/tutorial/async。如果我们有一个非常简单的异步函数如下。

#![allow(unused)]
fn main() {
// 假设func2() 也是async function, impl Future
async fn func1() {
    println!("func1");
    func2().await();
}
}

其实，async生成的匿名结构类似如下，是一个状态机。每次await都是一次状态的转移。

#![allow(unused)]
fn main() {
enum _func1 {
    State0,
    State1(func2),
    Terminate,
}

impl Future for _func1 {
    use _func1::*;
    type Output = ();
    fn poll(mut self: Pin<&mut Self>, cx: &mut Context<'_>)
        -> Poll<()>
    {
        loop {
        	match *self {
                State0 => {
                    println!("func1");
                    let future = func2();
                    *self = State1(future);
                }
                State1(ref mut future) => {
                    match Pin::new(future).poll(cx) {
                        Poll::Ready(out) => {
                            *self = Terminated;
                            return Poll::Ready(());
                        }
                        Poll::Pending => {
                            return Poll::Pending;
                        }
                    }
                }
                Terminated => {
                    panic!("future polled after completion");
                }
            }    
        }
        
    }
}
}

async fn 返回 Future ，而后者需要通过被不断的 poll 才能往前推进状态，同时该 Future 还能包含其它 Future ，那么问题来了。

  1. 谁来负责调用最外层 Future 的 poll 函数？下面会介绍的异步运行时会给出答案。
  2. 难道需要不停的poll吗？并不用。当第一次poll返回Pending之后，下一次poll将由回调函数wake触发。
  3. 这样子感觉还是串行，异步的优势如何体现？每一次返回Pending之后，都调度到其他线程上。

不过，在编程中，我们只需要一个.await，异步运行时就会自己帮我们去周期性的poll，在Pending之后调度到其他线程。

顺序模型

#![allow(unused)]
fn main() {
let x = a().await;
ley y = b(x).await;
}

当我们写下这样的代码的时候，函数a，b的执行还是串行的。不过有时候，可能b不依赖于a的结果，他们之间的执行顺序没有任何关系。这个时候，我们就有join!，他可以同时异步执行若干个任务，并等待所有任务执行完成。目前他还没有在标准库稳定下来，我们也可以通过第三方库future使用join。

#![allow(unused)]
#![feature(future_join, future_poll_fn)]

fn main() {
use std::future::join;

async fn one() -> usize { 1 }
async fn two() -> usize { 2 }

let x = join!(one(), two()).await;
assert_eq!(x, (1, 2));
}

或者，同时异步执行若干个任务，并等待所有任务执行完成，但是如果中间有一个任务出错了就立即返回，可以用try_join.

或者，我们想执行若干个任务，但是每次一个任务执行完成都会返回，可以用select。

Pin/Unpin

Unpin是一个标记特征，不需要我们实现。它表明一个类型可以随意被移动，默认情况下，Rust 中的所有类型都是可移动的。Rust 允许按值传递所有类型，以及常见的智能指针类型。

那之前很多地方出现了Pin类型，他是什么？它是一个智能指针。他的作用是，让指针指向的对象的内存位置不会移动。一个常见的场景是自引用结构，也就是一个包含了指向自己的指针的对象。显然，如果自引用结构的内存位置移动了，那么指向自己的指针就失效了，再去使用指针就会产生Undefined behavior。如下

#![allow(unused)]
fn main() {
// 我们需要把这个类型标记成Unpin，然后把他Pin住。
// 或者写Unsafe
struct SelfRef {
    name: String,
    p_name: *const String,
}
}

所以，Pin<P>类型的语义就是，让指针P指向的对象，有固定的内存位置，不会被移动，直到内存释放。我们说这个对象"pinned"。不过，Pin语义只对真的不能被移动的类型的指针有效，也就是被!Unpin标记的类型，比如Pin<&mut u8>，u8是被Unpin标记的，所以Pin<&mut u8>并不会产生效果。

tokio

显然，光有一个Future是远远不能实现异步执行的。要想真正发挥出异步的优势，我们要合理的调度Future。这便是**异步运行时(asynchronous runtime)**要做的工作。而tokio就是现在最主流、开发最活跃的异步运行时。不过，它的应用场景不包括CPU-bound和IO-bound的程序，在他的主页上提及。Tokio提供了相当多的异步支持，包括管道、IO、信号、网络等。

select!

#![allow(unused)]
fn main() {
select!{
    <pattern> = <async expression> , if <precondition> => <handler>,
    <pattern> = <async expression> , if <precondition> => <handler>,
    else => <expression>
}
}
  1. Evaluate all provided <precondition> expressions. If the precondition returns false, disable the branch for the remainder of the current call to select!. Re-entering select! due to a loop clears the “disabled” state.
  2. Aggregate the <async expression>s from each branch, including the disabled ones. If the branch is disabled, <async expression> is still evaluated, but the resulting future is not polled.
  3. Concurrently(not parallelly) await on the results for all remaining <async expression>s.
  4. Once an <async expression> returns a value, attempt to apply the value to the provided <pattern>, if the pattern matches, evaluate <handler> and return. If the pattern does not match, disable the current branch and for the remainder of the current call to select!. Continue from step 3.
  5. If all branches are disabled, evaluate the else expression. If no else branch is provided, panic.

当一个分支成功执行后返回，其余的分支被抛弃。所以所有的分支需要cancellation safe。
