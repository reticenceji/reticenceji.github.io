Reticence's Notes

Git

Basic Operation

这张图涵盖了git的基本操作

提交修改

首先将文件从工作区(workspace)提交到暂存区(staging area)。使用VSCode也可以在Changes的文件中点击+。

git add dir     # 将目录下的所有修改提交到暂存区
git add file    # 将文件的修改提交到暂存区

然后将暂存区的文件提交(commit)到本地仓库(local repository)。使用VSCode也可以直接填写Commit Message然后回车。

git commit -m "commit message"  
git commit --amend              # 不创建新的提交而是覆盖上一次提交

commit message可以参考BestPractice-CommitMessage

然后将本地仓库的修改推送(push)到远程仓库(remote repository)。

git push remote_respository_name local_branch_name:remote_branch_name

一般来说，只有一个远程仓库，命名为origin。主分支的名字一般为master/main，在开发的初期只需要一个分支，一般直接使用主分支。然后如果本地仓库和远程仓库的名字相同的话，也可以不写远程仓库名。

撤销修改

取消文件的暂存。使用VScode可以在Staged Changes中点击-号，进行的是第二种撤销。

git restore --staged file  # 从缓存区撤销到工作区
git reset --mixed file     # 从缓存区中删除，不改变工作区

取消工作区文件的修改

git resotre file

Advanced Operation

恢复文件

从某个版本中恢复指定文件

git checkout 830cf -- a.txt

从某个分支中恢复指定文件

git checkout master -- a.txt

查看文件修改历史

使用命令行

git log -p a.txt

使用VS Code的Gitlens提供的FILE HISTORY，会列出文件所有的修改记录。如果单击的话，会默认和上一次提交进行比较，不过第二个按钮是“Open changes with working file”可以和现在的文件进行比较。右边新，左边旧。

VSCODE文件视窗下的TIMELINE也是类似的。不过他提供了本地历史记录的功能，有时有用有时又有点乱。

分支切换

切换分支到master

git checkout master
git checkout master --discard-changes # 丢弃所有的改动
git switch master --discard-changes --recurse-submodules # 丢弃包括子模块的改动在内的所有改动
git switch master

创建并切换到新分支

git checkout -b new_branch_name
git switch -c new_branch_name

new_branch_name的命名可以参考 BestPractice-BranchNaming

可以看到在之前的恢复文件也是用checkout命令。正是因为checkout有着两个不同的用途，搞出了一个switch来专门用于在分支切换代替checkout。

子模块

在一个仓库里的子模块，对于父仓库来说可以认为是一整个文件，父仓库会保存子仓库的hash而不是文件，这样子模块发生了更改父仓库也可以感知到。

所以，在我们从远程拉下一个项目的时候，要去根据保存的hash去初始化子模块。

git submodule update --init --recursive   # 初始化本地模块，会从指定分支指定提交down下代码
git submodule update --remote   # 如果需要让本地模块和上游保持同步

子模块是很麻烦的，

版本回退

git reset --hard d85da # 注意，这个回退是不能后悔的

获取远程

其实git pull是一个复合命令。git pull = git fetch + git merge。如果只需要获取远程分支到本地而不执行merge，可以使用git fetch。他会修改ref/remote内到分支信息。

合并分支

合并分支有两种方法。rebase和merge。假设我们使用Github推荐工作流，在我们的feature开发一段时间后想要合并到main主分支，这时main已经有了一些其他同事的提交了，就像下面这样：

![A forked commit history](../images/01 A forked commit history.svg)

这时，我们想先将main的代码合并到我们的feature。可以使用merge

git merge feature main

merge会创建一个新的提交，把两边的改动合并。但是这样会有一个问题，你每次合入main的代码都会创建一个commit，没有实质作用，如果经常需要合入main就会很乱。

![Merging master into the feature branch](../images/02 Merging main into the feature branh.svg)

或者使用rebase

git checkout feature  # 切换到feature分支
git rebase main       # 自动合入main
git rebase -i main    # interactive合入main，可以进行更详细的配置

他的效果就像这样，会给feature上的每一个commit创建一个新的commit，附在main的后面，形成一个线性的提交。![Rebasing the feature branch onto master](../images/03 Rebasing the feature branch into main.svg)

如果使用interactive rebase，他会打开一个text editor，列出下面的信息供修改，你就可以合并提交、修改提交信息了。

pick 33d5b7a Message for commit #1
pick 9480b3d Message for commit #2
pick 5c67e61 Message for commit #3

差异比较

git diff 66ff c7b0 # compare commit c7b0 to commit 66ff

忽略文件

想要让git忽略一个文件，可以将它加入.gitignore

但是，一旦文件已经被追踪，在gitignore中加入就没有用了。这个时候我们就需要先将它取消追踪。

git rm --cached filename
git rm -r --cached dirname

Best PractiCe

Branch naming

一般通用的分支命名规范是：

  • feature/xxx: 新功能开发
  • bugfix/xxx: 修复缺陷
  • opt/xxx: 性能优化
  • infra/xxx: 工程化相关，例如构建脚本、配置文件的修改
  • chore/xxx:

xxx可以是简单的介绍，也可以带上自己的名字

Commit message

commit message的基本格式时 Header Body Footer

Header建议使用<type>:<scopt>/<subject>

  • type包括
    • feat: new feature
    • fix: fix bug
    • refactor: modifications other than features or fixes
    • release: release a new version
    • style: format or beautify code without effects on main function
    • chore(杂事): modifications of dependencies or build tools
    • docs: write documentation
  • scope介绍了commit修改的代码在哪个模块
  • subject介绍了修改的主要内容，建议使用动宾结构，不超过50个字符

Body包括了对修改的细节描述，这个可以分行的。

Footer一般没有，除非进行了breaking changes

示例

feat: 增加了xx功能

1. 增加了xx
2. 增加了xx

Work flow

Git早期的复杂工作流：Master，Develop，Feature，Release，Hotfix。

Github推荐工作流：只有一个主干分支，基于PR往主干分支提交代码。

Rebase Golden Rule

使用rebase的重要原则是，不能在公共分支上进行rebase。否则你的公共分支会和大家的都不一样，就乱了套了。rebase之后，往往需要force push。总之，rebase是一个危险的操作。

Git hook

在commit之前添加format/lint，在push之前进行单元测试，是推荐的做法。
