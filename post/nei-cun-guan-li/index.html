OS-Memory Management

2020-11-27 Views OS3174字13 min read繁

和厨房的垃圾桶一样，内存无论多大都是稀缺资源——CSAPP

物理地址

在计算机刚开始我们直接采用物理寻址
Variable——Externel Fragmentation。给进程分配可变大小的内存空间。问题是可能两个空间之间的间隙的内存太小，不足以容纳一个进程，造成浪费。
Fixed——Internel Fragmentaion。给进程分配固定大小的内存空间，问题是如果不能装载过大的进程，并且固定空间剩余没用的部分造成浪费。

虚拟地址/逻辑地址

CPU生成的地址称为逻辑地址，内存单元需要的地址是物理地址。他们之间的转换由MMU（Memory Management Unit）完成，也需要操作系统的配合。

Continuous memory

最容易想到的是把一个进程分配在一个连续的物理内存中，用base和limit两个寄存器。
我们检查Laddr<=limit，如果是的话Paddr = Laddr + base.。完成简单的保护。

segment

Segmentation的概念，主要是和limit。
逻辑地址由<segment, offset>组成。比如我们的text, data, rodata等。我们还可以对不同的段应用不同的权限保护。

Paging

Paging的关键是Page Table，说明了从 virtual memory映射到 physical memory的方法，顺便自然而然的完成了权限保护、内存共享。Paging实现的具体细节，可以参考下面的RISCV和Lab。

TLB

如果每一次访问内存都经过页表，那么我们要有n+1次对内存的访问（访问页表），这是十分低效率的。
为此我们引入了TLB，可以认为是对Page Table Entry的高速缓存（当然和cache有本质的区别）。
TLB能起作用，和Locality（局部性原理）是分不开的。

Frame

Free Frame的分配

我们加载一个进程、malloc的时候，都需要给进程分配空间。
像Virtual Memory Space的分配，这个比较方便。比如我们malloc就在堆中，加载进程就是那些段，.text，.data什么的。我们要讨论的是Frame的分配。
我们简单把所有的Frame看成一个Pool，向外界提供的接口是回收用完的Frame和提供Free Frame。具体的实现，可以是链表、buddy，bitmap等等。

我们可以给每一个Process一个frame pool，或者所有的进程一个global frame pool。
Global replacement – process selects a replacement frame from the set of all frames; one process can take a frame from another. Local replacement – each process selects from only its own set of allocated frames.
使用本地替换，分配给进程的页数是固定的，页替换只发生在分配给该进程的页之间。
在全局替换的情况下，任何页面都可能成为潜在的受害者，无论它当前是否属于寻求自由帧的进程。通常我们使用全局替换。

Reclaiming Pages

对于全局替换，我们引入一个水线的概念。
对于整个frame pool，我们不等到它空了再回收Frame。而是设置一个警戒水线，当free frame的数量低于水线的时候，就开始回收frame。
这个策略试图保证我们总是有充足的Free frame来满足请求。

Swap

如果物理Frame都用完了，但是我们还需要给Page分配Frame，这个时候就要swap一个物理的Frame到Disk上，然后再把这个Frame分配给Page。
下面举例，我要从Disk上加载一个Page：

  1. 首先，在Disk上找到要加载的Page。
  2. 在物理内存中找一个Free Frame(当然不是遍历物理内存，操作系统会维护Free Frame)。
  3. 如果找到了，就用它。
  4. 找到一个Victim Frame。如果victim Frame是Dirty的，需要写回到Disk上。
  5. 现在我们得到了Free Frame。更新Page Table。
  6. 返回发生Trap的指令继续执行。
    那么如何选择这个Victim Frame呢？我们有很多策略。

FIFO（简单）

First in First out。虽然简单但是效果不怎么理想。

LFU

Least Frequently Used（不实用）

MFU

Most Frequently Used（不实用）

Optimal Algorithm（理想）

假设我们知道哪个Page在未来最后被使用（不用的时间最长），就把这个Page kick out。
和Shortest Job first一样，虽然最好，但是要求上帝视角，所以只是一个理论模型。

LRU=Least Recently Used（实用）

Associate time of last use with each page, select pages with oldest timestamp。选择一个最久没有被使用过的Frame置换。
问题是，如何确定一个最久没有被使用的Frame呢？具体的实现有：

  1. Counter Based(不实用)：为每个PT Entry关联一个存储时间的结构。这代价有点高。
  2. Stack(不实用)：当每个Frame被使用的时候把他移动到Frame堆栈的顶端，这样子最久没有被使用过的Frame就在堆栈的底端。这代价也很高。
  3. reference bit的近似：我们在PTE中有一个R bit，可以用来标记Entry是否被Refer了。近似的实现使用时间的标记。
    1. 8 bit的方式：我们用8个bit标记时间。每个时间单元我们都会把entry的R byte右移一位，然后把有使用的Entry的最高位置1。这样，通过把R byte作为一个无符号数进行比较，最小的就是最久没有被使用的。
    2. 第二次机会：我们选择一个页面时，如果R值为0就选择，为1的话置0再继续寻找。
    3. enhance second chance：我们用Refered bit 和 Modified bit来标记。这两位就像读和写一样。
      (0,0) 最近没有被读也没有被修改，最佳选择
      (1,0)最近被读没有被修改，不用写回但可能很快就要被使用
      (0,1)最近被修改但是没有被读，需要邪回
      (1,1)最近被读写，最不好的选择

Linux & page fault

mm_struct, switch_mm,mm_struct.pgd
有两个结构，一个是页表，一个是vm_struct.
平时我们通过页表访问，页表的标志位帮我们实现保护。

当出现page_fault的时候，我们才会遍历vm_struct(vma在Linux中用红黑树来加速查找)，检查是为什么出现了page_fault。
如果不是valid的，可能是非法指令、段错误等等。
如果是valid的，那应该是被swap到Disk里了。

这个和Demand Paging好像差不多啊。从这个例子中看，都是invalid惹得祸。只不过Lasy allocation只需要分配一个free frame，demand paging需要从Disk中加载内容到一个free frame。

copy on write——fork

我们在Fork的时候，需要复制Parent的内存内容，但是大多数情况我们会exec刷掉复制的内容。所以我们引入copy on write技术防止这个复制过程的浪费。

  1. Allowing the parent and child processes initially to share the same
    pages. These shared pages are marked as copy-on-write pages, meaning that
    if either process writes to a shared page, a copy of the shared page is created.如果我们试图要写的哪一块的话，再创建一个副本。
  2. 还有vfork系统调用。vfork并不使用copy on write技术，就是单纯的和Parent指向了同样的frame。所以我们要保证之后立即调用exec，不修改父进程的内容，否则就出问题了。

Buddy(曾经的Linux)

用于Kernel的Frame管理。避免External Fragmentation。

slab,slob,slub

用几个物理连续的页，防止Internel Fragmentation。Slab还可以作为一个pool，比如task_struct_slab，统一管理task_struct用的Frame。

抖动thrashing

频繁的发生page fault。一个进程没有足够的Frame使用，那么就会频繁的产生Page Fault，在Memory和Disk上反复切换，造成极大的性能问题。并且此时CPU使用率下降，CPU会误以为空闲并增加多进程程度，恶性循环。
怎么解决抖动问题？

working-set

引入一个working-set的概念。我们用δ\deltaδ表示working-set window，最近δ\deltaδ个页面引用的页面集合称为working-set。我们用working-set来检测Locality。

这样子，如果我们为进程分配的Frame<working-set，我们就给他分配更多的Frame。如果没有更多的Frame了，就只能挂起别的Process了。
对Locality的检测取决于working-set的大小，太小了不足以包含整个Locality，太大了可能包含多个Locality。

PFF(Page-Fault Frequency)

If the actual page-fault rate exceeds the upper limit, we allocate the process another frame. If the page-fault rate falls below the lower limit, we remove a frame from the process.

逻辑地址=>物理地址


S 模式提供了一种传统的虚拟内存系统，它将内存划分为固定大小的页来进行地址转
换和对内存内容的保护。上面我们介绍了satp,比如我们开启了Sv39转换。

为什么是39 = 12+9+9+9呢？
因为1Page = 4Kb，并且我们是64位CPU。这是两大前提。
VPN是9位，因为64bit = 8byte，那么1Page可以容纳的Entry数量就是512 = 2^9，我们的VPN是用来在page table作为index的，所以就是9位啦。
最后为啥是12bit？ 4k=2^12。

我们的虚拟地址需要39bit，但是我们每一个页表的entry还是64bit。为什么呢？
首先是为了对齐，我们是按照byte处理的，39不伦不类效率很低。
那么为什么不是40呢？其实这样子还有一个扩展性的考虑。个人理解。

物理地址为啥是这样呢？为啥不和虚拟地址统一呢？
emmm,我觉得可以，但是没必要。物理地址支持的范围不是多多益善嘛。
只不过对于某个进程来说，就算你的物理内存空间真的把物理地址填满了，进程也用不了那么多的内存。虚拟地址是上限了。
当然，这是用户的进程。对于内核来说，他可是可以更改自己虚拟地址的映射的。通过改变page table可是能把所有物理地址都用上。

Kernel和User

来看看Linux对内存映射的分布。我们知道，随着进程的切换，页表也要相应的改变，从而给每一个进程一个独立的地址空间。但是Kernel的映射是不变的。
所以，我们把内存映射（页表决定的逻辑地址的分配）分成两部分，一部分是不变的Kernel，另一部分是变化的User 。
