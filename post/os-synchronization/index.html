OS-Synchronization

2020-11-10 8 min read # OS

首先我们需要明确，我们需要解决的问题：Critical Section Problem，Critical section同时只能有一个进程访问，否则可能会得到错误的结果。

如果禁用中断，可以避免kernel的critical section问题。但是这显然不好。

然后明确怎么样算解决了这个问题，我们的算法要满足三个条件：

  1. Mutual Exclusion：互斥。
  2. Progress：总有一个进程在执行。
  3. Bounded waiting：不会有一个一直无法得到执行的进程。

Peterson’s Solution

这个方法适用于两个进程交错执行Critical Section和Remainder Section，并假设语句是原子的。

int turn;	//下一个执行哪个进程
boolean flag[2];	//flag[i]==true表示Pi已经准备要执行了
while (true) {
	flag[i] = true;
	turn = j;
    //意思是，如果另一个进程没有准备好，或者现在轮到我执行了，我就执行
	while (flag[j] && turn == j);
		/* critical section */
    //我不准备执行了
	flag[i] = false;
		/*remainder section */
}

分析他是一个 solution to the critical-section problem，不严谨的分析。

  1. Mutual exclusion：显然，如果flag[i]==false，那么他要么是执行过了，要么还没准备要执行；如果flag[i]==true and flag[j]==true，那么turn将决定执行谁。（我们假设这里的每条句子是原子的）最后turn是谁就执行谁。
  2. **Progress：**我们注意到Pi不执行的情况当且仅当flag[j] && turn == j，Pj不执行的情况当且仅当flag[i] && turn == i，他们不会同时满足，总有一个成立。
  3. **Bounded waiting：**因为就两个进程，每个进程Pi执行完成后设置flag[i]=false，此时另一个等待的进程满足执行条件。

低级的原子语句

这是解决Critical Section算法的基础。

test_and_set Instruction

假设硬件实现了对test_and_set的原子实现。下面是伪代码：

boolean test_and_set(boolean *target) {
    boolean rv = *target;
    *target = true;
    return rv;
}//返回值是true表示被加lock了

一个利用test_and_set的小例子：

do {
    while (test and set(&lock));
    	/* do nothing */
    	/* critical section */
    lock = false;
    	/* remainder section */
} while (true);

lock==true表示占用了资源

  • 如果lock之前是true，那么说明资源被别的程序占用了，我们进不去。
  • 如果lock之前是false，在test_and_set中，我们实现了lock=true和返回false的操作，我们开始了critical section并且加了锁，防止其他程序进入。

compare_and_swap

int compare_and_swap(int *value, int expected, int new value) {
	int temp = *value;
	if (*value == expected)
		*value = new_value;
	return temp;
}

一个利用compare_and_swap的小例子：

while (true) {
    while (compare and swap(&lock, 0, 1) != 0)
        ; /* do nothing */
        /* critical section */
    lock = 0;
   		/* remainder section */
}

和上面类似的，lock==1表示资源被锁住了。

  • 如果lock之前是0，表示没有上锁，lock==expected，我们加上锁然后返回0。
  • 如果lock之前是1，表示已经上锁，lock!=expected，我们返回1。

上面的两个小例子虽然能做到MUX，但是这样不 satisfy the bounded-waiting requirement。为什么呢？因为可能出现两个进程相互交替然后其他进程无法执行的情况。来看下面这个例子：

boolean waiting[n];
int lock;
while (true) {
    waiting[i] = true;
    key = 1;
    while (waiting[i] && key == 1)
    	key = compare_and_swap(&lock,0,1);
    waiting[i] = false;		//我不等了，防止下次还执行到我。
    	/* critical section */
    j = (i + 1) % n;
    while ((j != i) && !waiting[j])
    	j = (j + 1) % n;	//挑选下一个准备执行正在waiting的process
    if (j == i)	
    	lock = 0;	//没找到在等的，把锁释放了，不然就死锁了。
    else
    	waiting[j] = false;		//找到在等的，就让他不等啦，下一个执行的就是他。锁被传递给他。
    	/* remainder section */
}

这里的waiting表示已经准备执行了，wating[i] = false表示已经不再等待了。

key表示能否获得"钥匙"。就是能否使用资源，key=0表示可以。

注意到这里的扫描机制，每个等待的进程在n个进程之内必然被执行。如果i执行了之后，必须其他正在等待的都被执行了才能轮到i。test_and_set和compare_and_swap是类似的，也可以实现上面的功能。

注意到上面提到的三个例子都是busy waiting，会浪费很多的系统资源，尤其是很多进程的时候。我们叫他自旋锁（SpinLock）。

Mutex 互斥锁

这是最简单的工具了。

每个Mutex有一个布尔变量available主要有两个操作（avilable=1表示可用，假设下面的操作是原子的）：
简单的可以这样实现，也是spinlock。

acquire(){
	while(compare_and_swap(avilable,1,0));
}

release(){
	compare_and_swap(available,0,1);
}

Semaphore 信号量

semaphore比mutex功能更强大，主要也是两个操作，wait和signal。
wait() 等待一个被释放的锁然后占用。S->value>0表示没有进程在等待资源，锁没有被占用。S->value<=0表示被占用。下面这两个实现解决了忙等待问题。

wait(semaphore *S) {
    S->value--;	//等待的人+1
    if (S->value < 0) {		//得不到锁
    	add this process to S->list;
    sleep();
	}
}//linux中这个函数叫down()

signal(semaphore *S) {
    S->value++;	//等待的人-1
    if (S->value <= 0) {	//如果有在等的，就可以执行了。但是锁还是占用状态，相当于是把锁继承给他了
    	remove a process P from S->list;
    wakeup(P);
	}
}//linux中这个函数叫up()

需要注意的是，上面的两个操作应该是原子的。所以在Linux内核的代码中，有一个spin_lock来保证这些操作是原子的。spinlock还是被使用，但是里面的代码很短。

Priority Inversion、Starve、Deadlock

Priority Inversion：一个低级别的进程占用了锁，让高级别进程无法执行。我们用Priority inheriance 解决这个问题。
Starve：虽然整体仍然在继续进行，但是有个倒霉的进程得不到执行。
Deadlock：因为不合理的设计让几个进程相互等待然后都得不到执行，停滞了。

经典同步问题

Bounded Buffer Problem/Producer-Consumer Problem

缓冲池有n个buffer，Producer为Consumer生产满的buffer，Consumer为Producer生产空的buffer。他们共享数据结构如下，我们要实现的目标是他们不能同时使用buffer，Producer必须在不full的时候才能生产，Consumer必须在不empty的时候才能消耗。

int n;
semaphore mutex=1;
semaphore empty=n;
semaphore full=0;

解决方案如下：

//Producer
do {
    //item produce
	wait(empty);
	wait(mutex);	//注意这里的顺序是不能反的。
	
    //add item to buffer
    
    signal(mutex);
    signal(full);
while (1);

//Consumer
do{
    wait(full);
    wait(mutex);
    
    //remove item from buffer
    signal(mutex);
    signal(empty);
    
    //use item
}while(1)

Readers-Writers Problem

可以很多读者一起读，但是只能有一个作者在写。读写互斥。使用数据结构如下

semaphore rw_mutex=1;
semaphore mutex=1;
int read_count;

解决方案如下：

//writer
do{
	wait(rw_mutex);
	/* writing */
	signal(re_mutex);
}

//reader
do{
	wait(mutex);
	read_count++;
	if (read_count==1)
		wait(rw_mutex);		//第一个读者应该把读写锁置上
   	signal(mutex);	//mutex的作用是为了让这一段是原子操作
    
    wait(mutex);
    read_count--;
    if (read_count==0)
        signal(re_mutex);
    signal(mutex);
}

如果不停的有读者在读，writer就starve了。这是reader优先的做法。

另一个是writer优先的做法，当有writer在等待的时候，新来的读者被suspend了。

Dining-Philosophers Problem


这是一个有趣的问题，一个解决方法是单号哲学家得先拿到左边的筷子才能拿右边的筷子，双号要先拿到右边的筷子才能拿左边的筷子。
