ADS-External sorting

2021-06-23 Views DS1179字6 min read繁

主要是为了解决很多数据以至于内存中放不下的情况。我们学过的Merge sort就是一种external sort。

评价普通的排序只需要考虑时间复杂度，评价externel sort要考虑

  1. seek time O(pass)O(pass)O(pass) 寻道次数
  2. time to read or write one block of records
  3. internal sort M records
  4. merge N records from input buffer to output buffer

基本的external sort思想

假设我们的内存能容纳M个数据，我们一共有N个数据存储在Tape 1上。

  1. 依次读取M个数据，在主存中进行排序。然后分别写入Tape 2和Tape 3。从这时候开始，数据被分成了N/M个有序的顺串。
  2. 然后合并T2+T3=>T1+T4， T1+T4=>T2+T3。每次我们都可以将块的数量/2
  3. 直到两边块的数量都是1,然后合并到一个Tape上。排序结束。

所以我们不难发现，pass = 1+log2(N/M)1+log_2(N/M)1+log2​(N/M)。1指的是第一次读取全部的数据进行排序，log2(N/M)log_2(N/M)log2​(N/M)指的是后面归并的过程。

k-way merge

可以减小pass，减小seek次数。上面的其实就是2-way merge。对于k-way merge ，pass=1+logk(N/M)pass=1+log_k(N/M)pass=1+logk​(N/M)。

过程：以3-way 为例，从3个磁盘中读取出一个数，维护一个（我觉得MinHeap也可以？不知道为什么是）Loser tree。Tree的根节点一定是最小的元素，把它写出来，读一个新的进去。如此反复。

In general, for a k-way merge we need 2k input buffers and 2 output buffers for parallel operations.

Polyphase Merge

上面提到的k-way Merge 需要2k个磁带，事实上我们只使用K+1个磁带也可以完成工作。比如下面的2路归并排序，我们只需要三个磁盘，按照斐波那契数列的规则把初始数据分到磁盘上。可以通过填充的方式变成斐波那契数个数据。

Replacement selection

我们之前构造顺串的做法是，读取M个元素，在内存中排好序，输出到磁带。所以我们会得到N/MN/MN/M个顺串。但其实这个步骤是可以优化的，每次把一个记录写回磁盘之后，又可以再读入新的数据。

T[1] = {81,94,11,96,12,35,17,99,28,58,41,75,15}
如果按照标准分法3-way merge，会是
T[2] = {11,81,94;	41,58,75;}
T[3] = {12,35,96;	15;}
T[4] = {17,28,99;}

        H[1]   H[2]     H[3]       OUTPUT          NEXT
---------------------------------------------------------------
Run 1    11     94       81          11             96			读入的数据比序列的最后一个小，就有机会写回
         81     94       96          81             12*			读入的数据比序列的最后一个大，就Dead了
         94     96       12*         94             35*
         96     35*      12*         96             17*
         17*    35*      12*       End of Run.  Rebuild Heap	当Heap中全是Dead，就需要重新一轮构建顺串了
---------------------------------------------------------------
Run 2    12     35       17          12             99
         17     35       99          17             28
         28     99       35          28             58
         35     99       58          35             41
         41     99       58          41             15*
         58     99       15*         58         end of tape
         99              15*         99
                         15*       End of Run.  Rebuild Heap	当然，如果数据读完了，一轮也就结束了
---------------------------------------------------------------
Run 3    15                          15
            
所以我们得到
T[2] = {11,81,95,96;}
T[3] = {12,17,28,35,41,58,75,99;}
T[4] = {15;}

通过这种方式得到的顺串，平均长度是2M。

这样得到的串的长度就不是相等的了，有什么好办法减小顺串的Merge time呢？答案是Huffman tree。
