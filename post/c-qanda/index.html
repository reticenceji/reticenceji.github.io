什么时候应该用引用作为参数，是不是能用引用的地方都可以用指针？什么时候应该用原值？引用作为返回值？

一个 reference 通常是以一个指针来实现的，所以 point 和 reference 并没有本质的区别。（那下面这个i*3怎么解释）

引用可以是l-value也可以是r-value。

func(const double& a)：如果不想更改a，那么这是最好的办法。而且这样子a就只能是一个r-value。下面的i*3就合法了。

func(double& a)

如果要用func(i*3)，我们就不能用引用或者指针了，因为i*3不是一个左值

RTTI是什么？

RTTI(Run Time Type Identification)即通过运行时类型识别，程序能够使用基类的指针或引用来检查着这些指针或引用所指的对象的实际派生类型。
一般来说，每一个 class 相关联的 type_info 对象的指针 通常也保存在 vtable 的第一个 slot 中。

多态和虚函数是怎么实现的？

C++在内存布局以及存取时间上的主要额外负担都是由 virtual 引起的，包括：

  • virtual function 机制：引起了保存 vtable 和透过 vtable 找到函数地址；
  • virtual base class 机制：引起了透过指针来找到基类的成员。

Virtual function 机制则由以下的 2 个的两个步骤来支持：

  • 每一个 class 产生出一系列指针 Virtual function 的指针，放在一个被称为 virtual
    table(vtbl, vtable)的表格中。
  • 每一个 class object 被添加了一个指针 vptr，指向相对应的 vtable。vptr 的设置由编译器
    全权负责，程序员无需关心。
    一个 vtable 对应 一个 class ， 一个 vptr 才对应 一个 class object

用GDB来调试以下程序：

#include <iostream>

class Base{
    private:
        int i;
    public:
        Base():i(10){std::cout << "I'm Base()" << std::endl; }
        ~Base(){std::cout << "I'm ~Base()" << std::endl; }
        virtual void print(){std::cout << "I'm base_print()" << std::endl;}

};

class Derived : public Base{
    private:
        int i;
    public:
        Derived():i(12){std::cout << "I'm Derived()" << std::endl; }
        ~Derived(){std::cout << "I'm ~Derived()" << std::endl; }
        void print(){std::cout << "I'm derived_print()" << std::endl;}
};

int main() {
    Base base = Base();  
    Derived derived= Derived();
    base.print();
    derived.print();	
    std::cout << sizeof(Derived) << std::endl;
    return 0;
}

struct 和 class 的区别？

在语言层面并无本质的区别，更多的是概念和编程思想上的区别。
struct 用来表现那些只有数据的集合体 POD（Plain OId Data）、而 class 则希望表达的是 ADT（abstract data type）的思想。
由于这 2 个关键字在本质是无区别，所以 class 并没有必须要引入，但是引入它的确非常令人满意，因为这个语言所引入的不止是这个关键字，还有它所支持的封装和继承的哲学。

什么是trivial和non-trivial?

trivial意思是无意义，这个trivial和non-trivial是对类的四种函数来说的：
构造函数(ctor)
复制构造函数(copy)
赋值函数(assignment)
析构函数(dtor)

如果至少满足下面3条里的一条，那么上面的四种函数是non-trivial函数：

  1. 显式(explict)定义了这四种函数。
  2. 类里有非静态非POD的数据成员。(比如有virtual函数)
  3. 有基类。

如果这个类都是trivial ctor/dtor/copy/assignment函数，我们对这个类进行构造、析构、拷贝和赋值时可以采用最有效率的方法，不调用无所事事正真的那些ctor/dtor等，而直接采用内存操作如malloc()、memcpy()等提高性能，这也是SGI STL内部干的事情。

换句话说，虽然那些函数在语义上存在，但不会被真正产生出来。比如拷贝，我们可以用memcpy直接 bitwise copy semantics。

在成员初始化列表背后实际发现的事情是什么呢？
  • 初始化成员：使用 member initialization list 或者调用默认构造函数；
  • 在那之前，如果 class object 有 vptr，它们必须被正确的设置；
  • 在那之前，所有的上一层的 base class construcotrs 必须被调用，以 base classes 声明
    的顺序。使用 member initialization list 或者调用默认构造函数，同时如果 base class
    是多重继承下的非第 1 基类，还需要调整 this 指针；
  • 在那之前，所有的 virtual base class constructors 必须被调用，从左到右，从深到浅。
    并同时设置好 virtual base class 所需要使用的各种机制；

即处理顺序为：virtual base classes → base class → vptr → member。

编译器会一一操作初始化列表，把其中的初始化操作以 member 声明的次序在 constructor
内安插初始化操作，并且在任何 explicit user code 之前。

以 member 声明的次序来决定初始化次序和初始化列表中的排列次序之间的外观错乱，可
能会导致一些不明显的 Bug。

不过 GCC 已经可以强制要求使用声明次序来进行初始化以避免了这个陷阱。

x.x 和 px->x 有什么区别？

X x; x.x = 0.0;和X *px; px->x = 0.0;
这两种写法在什么时候有重大的区别呢？
答案是当类型 X 是一个继承体系中有虚基类的子类，并且 x 成员又正好是虚基类中的成员时，这两种写法在编译器看来就会有重大的区别了（这种情况下的 offset 计算方式不同，其它情况的offset 也可以直接算出）。因为 x 的类型是固定的，编译器可以直接向第 8 条中所述的进行offset 扩展，但是对于 pt 来说，由于无法确定 pt 指向的真正类型，所以只能借由一个执行期的间接来得到成员的具体地址。

x.x是一个静态绑定，不会有多态的特征。

成员函数在哪里？

普通函数、普通成员函数、静态成员函数到最后都会变成与 C 语言函数类似的普通函数，只是编译器在这些不同类型的函数身上做了不同的扩展，并放在不同的 scope 里面而已。也就是说，都在.text段中。

Type1 X::foo(Type2 arg1) { … }
会被转换为如下的普通函数（应该会改变函数的名字，变成奇奇怪怪的样子）：
void foo(X *const this, Type1 &__result, Type2 arg1) { … }返回值都没了？那对应的汇编是什么鬼样子，如果我把他当作一个过程调用会怎么样？

纯虚函数是什么？

纯虚函数是一种特殊的虚函数，在许多情况下，在基类中不能对虚函数给出有意义的实现，而把它声明为纯虚函数，它的实现留给该基类的派生类去做。这就是纯虚函数的作用。

C++中的纯虚函数，一般在函数签名后使用=0作为此类函数的标志。但是纯虚函数也可以有实现，但是有纯虚函数的class不能被实例化，所以这个实现应该是给子类使用的。

什么时候会发生Copy？Copy的时候发生了什么？

Base::Base(const Base& a) Copy的函数原型。C++会提供默认的Copy，单纯的把内容复制。但是一旦复制的内容里有指针，可能默认的Copy就不是你的本意了。

void func (Base x) {...};
Base func2 () {Base a; ... return a;};
Base a;			//这样会调用一个Constructor
Base b = a;		//这样会调用一个Copy Constructor
Base b(a);		//这个和上一句是一个意思，也会调用Copy Constructor
func(b);		//作为参数传递，也会调用一个Copy Constructor
func2();		//作为返回值，也会调用一个Copy Constructor

虽然上面这么说，但是编译器可能会优化掉一些没有必要的Copy。

如果有指针，默认的Copy动作会单纯的复制你的指针，这样a和b就中的指针就指向了同一个内存区域。可能会造成什么问题呢？

  1. 可能会对同一块内存析构两次。
  2. 可能会修改我本来不想修改的内容。
