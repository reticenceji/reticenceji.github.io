Reticence's Notes

Sematic Analyzing

https://www.yuque.com/xianyuxuan/coding/cp_4

在一个典型的静态类型的语言 (如C语言)中，语义分析包括构造符号表、记录声明中建立的名字的含义、在表达式和语句中进行类型推断和类型检查以及在语言的类型规则作用域内判断它们的正确性。

这里提到的语义分析是 静态语义分析，不包含可能的逻辑错误，如死循环、除以 0 等情况。

例如，标识符的属性就包括其种属（变量、数组、函数名等等）、数据类型、存储位置、长度、值、作用域等。

属性的确定时间是有多种可能的。属性值的计算以及将计算出来的值与相关的语言结构进行联系的过程称为属性的 绑定binding，发生的时间称为 binding time。在程序执行前（编译过程中）进行 binding 的属性称为 static attributes；在程序执行过程中才进行 binding 的属性称为 dynamic attributes。在编译过程中进行语义分析的时间也是有多种可能的，我们可以在语法分析生成完AST之后再进行语义分析，这就是遍历一遍语法树；也可以将语法分析和语义分析结合在一起。

属性计算

我们通过为CFG中的文法符号设置语义属性（简称属性），来表示语法成分对应的语义信息。属性可以是任何类型的，比如numbers, types, table references, or strings。文法符号的语义属性值是用与文法符号所在产生式（语法规则）相关联的**语义规则(semantic rule)**来计算的。比如下面这个产生式。我们就可以给文法符号digit的value属性赋值成0。digit.value = 0。

digit -> 0

语法制导定义(Syntax-Directed Definitions, SDD)是对CFG的推广，也叫属性文法(Attribute Grammar)（不过他们还是有区别的，属性文法要求没有副作用）

  • 将每个文法符号和一个语义属性集合相关联。
  • 将每个产生式和一个语义规则集合相关联，这些规则用于计算该产生式中各文法符号的属性值。

**语法制导翻译方案(Syntax-Directed Translation Scheme, SDT)**是在产生式右部嵌入了程序片段的CFG，这些程序片段称为语义动作。按照惯例，语义动作放在花括号内。

综合属性和继承属性

属性根据不同的计算方式，分成两种：

  • A synthesized attribute(综合属性) for a nonterminal A at a parse-tree node N is defined by a semantic rule associated with the production at N. Note that the production must have A as its head. A synthesized attribute at node N is defined only in terms of attribute values at the children of N and at N itself. 另外，终结符从词法分析器处获得的属性值被归为综合属性值
  • An inherited attribute(继承属性) for a nonterminal B at a parse-tree node N is defined by a semantic rule associated with the production at the parent of N. Note that the production must have B as a symbol in its body. An inherited attribute at node N is defined only in terms of attribute values at N's parent, N itself, and N's siblings.
  • 给定一个属性文法，通过适当地修改文法，而无须改变文法的语言，所有的继承属性可以改变成综合属性。

从上面两种计算方式不难看出，AST上节点的属性计算依赖于其他节点。我们可以把这种依赖关系表达为有向图的形式，称为依赖图。依赖图的节点是语法分析树的节点，有向边是从被依赖节点指向节点。在依赖图中，综合属性表现为边只从子节点指向父节点。

我们的依赖图是不应该出现环的，因为环意味着循环依赖，依赖图应该是有向无环图我们才好正确的计算每个节点的属性。对于一个无环图，我们可以按照拓扑排序的顺序进行属性求值。

S-SDD和L-SDD

虽然我们可以按照拓扑排序的方式去寻找一个合理的求值顺序（或判断是否存在合理的求值顺序），不过在实际运用上，我们有两个实用的无环图的子类：

  • S-属性定义(S-SDD)。仅仅使用综合属性的SDD。他可以在bottom-up parsing的时候计算，或者在top-down parsing的时候采用post-order的方式遍历进行计算。

  • L-属性定义(L-SDD)。直观含义是在一个产生式所关联的各属性之间，依赖图的边可以从左到右，但不能从右到左。严格的说，一个SDD是L-SDD，当且仅当它的每个属性要么是一个综合属性，要么是满足如下条件的继承属性：假设存在一个产生式$A→X_1X_2…X_n$，其右部符号$X_i(1 \le i \le n)$的继承属性仅依赖于下列属性：

    1. $A$的继承属性
    2. 产生式中$X_i$左边的符号 $X_1, X_2, … , X_{i-1}$ 的属性
    3. $X_i$本身的属性，但$X_i$的全部属性不能在依赖图中形成环路

    所以S-SDD是L-SDD的一种特殊情况。

S-SDD在依赖图中，表现为总是通过子结点的属性值来计算父结点的属性值。可以和自顶向下、自底向上的语法分析过程一起计算。并且如果一个S-SDD的基本文法可以使用LR分析技术，那么它的SDT可以在LR语法分析过程中实现。

  • 自底向上：在构造分析树的结点的同时计算相关的属性（此时其子结点的属性必然已经计算完毕）
  • 自顶向下：递归子程序法中，在过程A()的最后计算A的属性（此时A调用的其他过程（对应于子结构）已经调用完毕）
#![allow(unused)]
fn main() {
// 自底向上伪代码实例
enum Action {
    Shift, 
    Reduce(u32), 
    Error(&'static str),
    Accept
}
// 我们假设所有Token的属性都已经计算好了
fn parse(input: &[Token]) {
    let mut state = 0;       
    let mut stack = vec![];  
    let mut input = input.iter();
    loop {
        let token = input.next().unwrap();
        match ACTION(state, token) {
        	Action::Reduce(pr) => {
                let (A, beta) = PRODUCTION[pr]; 
                for _ in 0..beta.len() {        
                    let token = stack.pop().unwrap();  // AST上节点的所有子节点都被遍历
                    A.compute(&token);                 // 把子节点的属性拿进去计算
                }
                stack.push(A);
                state = GOTO[state][A];
                ...
            },
        	Action::Shift => {
                stack.push(token); 
            },
            Action::Accept => break,
        	Action::Error(e) => panic!("Error: {}",e),
    	}
    }
}
}

L-SDD稍微复杂一点，将L-SDD转换为SDT，需要：

  1. 将计算某个非终结符号A的继承属性的动作插入到产生式右部中紧靠在A的本次出现之前的位置上。
  2. 将计算一个产生式左部符号的综合属性的动作放置在这个产生式右部的最右端。

所以我们怎么实现他呢？如果一个L-SDD的基本文法可以使用LL分析技术，那么它的SDT可以在LL或LR语法分析过程中实现。

  • 在自顶向下中实现L-SDD，对于每个非终结符号A，其对应的过程的参数为继承属性，返回值为综合属性
  • 在自底向上中实现L-SDD，对于每个非终结符号A，计算他的合成属性时，可以直接利用还存在在栈上的数据的属性，在计算他的继承属性的时候，往往只能存在一个外部数据结构上。

有时，我们不好将属性作为函数的参数和返回值进行传递，这时，我们可以

  1. 将属性保存在syntax tree上。
  2. 将属性保存在external data structure中，比如符号表。

符号表

在编译器中符号表是一个典型的目录数据结构。插入、查找和删除这3种基本操作。典型实现包括线性表、各种搜索树结构 (二叉搜索树、AVL树、B树)以及Hash表等。

通常使用Hash表并采用separate chaining的方式解决冲突，因为他的插入、删除和查找的时间代价都差不多并且比较低。如下图所示

有一系列可能的方法可以实现嵌套的作用域

  1. 为每个作用域建立一个新的符号表，再从内到外把它们链接在一起，这样如果查找操作在当前表中没有找到名字，就自动用附上的表继续搜索。这样的话，退出作用域会非常方便，只需要清理掉新创建的符号表就行了。

  2. 符号表插入操作不必改写前面的说明，不过查找操作只能找到名字最近插入的说明；类似地，删除操作不应删除与这个名字相应的所有说明，只需删除最近的一个。

类型检查

类型检查，如果两边的类型不匹配可能会发生隐式转换（弱类型），也可能报错（强类型）。

首先，我们需要基本类型Basic Types，比如C语言里的int和double。那么类型构造器Type Constructor就是一个函数，他以类型为输入，输出另一个类型。C语言中的类型构造器有下面几种：

  • 数组/指针（C语言里是一样的）：比如int *a，a的类型是int *，a[1]的类型是int。

    TC(int *) = int

  • 结构体：比如struct {double d; int i;} S构成了一个新的类型。

    TC(double, int) = struct S

  • 联合：比如union {double d; int i;} U构成了一个新的类型。

    TC(double, int) = union U

  • 函数：比如int f(double d, int i);构成了一个新的类型。

    TC(int, double, int) = int(*f)(double, int)

我们什么时候认为两种类型是等价的？由弱到强

  • structure equivalence: two types are the same if and only if they have the same structure.
  • declaration equivalence: weaker version of name equivalence( t2 = t1 are interpreted as establishing type aliases, rather than new types).
  • name equivalence: two type expressions are equivalent if and only if they are either the same simple type or are the same type name

比如，我们定义两个类型，以下为Rust代码

#![allow(unused)]
fn main() {
struct Meter(u32);
struct KiloMeter(u32);
type Mi = Miter;
}
  • 如果我们认为Meter和KiloMeter有一样的类型，那么就是structure equivalence。
  • 如果不认为Meter和KiloMeter有一样的类型，而且不认为Meter和Mi有同一类型，那么就是name equivalence。
  • 如果不认为Meter和KiloMeter有一样的类型，不过认为Meter和Mi有同一类型，那么就是declaration equivalence。

这么看来，Rust应该是declaration equivalence。
